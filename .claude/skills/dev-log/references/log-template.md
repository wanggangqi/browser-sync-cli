# 开发日志模板

## 标准格式

```markdown
# {{date}} 开发日志

## [{{change_type}}] {{title}}

| 字段 | 值 |
|------|-----|
| 时间 | {{time}} |
| 类型 | {{change_type}} |
| 影响文件 | {{files}} |
| 标签 | {{tags}} |

### 变更描述
{{description}}

### 变更原因
{{reason}}

### 测试验证
{{test_notes}}
```

---

## 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `{{date}}` | ✅ | 日期，格式：YYYY-MM-DD |
| `{{time}}` | ✅ | 时间，格式：HH:MM |
| `{{change_type}}` | ✅ | 变更类型（见下方列表） |
| `{{title}}` | ✅ | 简短标题 |
| `{{files}}` | ⚠️ | 影响的文件列表（如有） |
| `{{tags}}` | ⚠️ | 标签列表（建议填写） |
| `{{description}}` | ✅ | 变更详细描述 |
| `{{reason}}` | ⚠️ | 变更原因/背景 |
| `{{test_notes}}` | ⚠️ | 测试验证说明 |

---

## 变更类型列表

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| `feature` | 新功能 | 添加新功能模块 |
| `bugfix` | Bug 修复 | 修复已知问题 |
| `refactor` | 重构 | 代码重构，不影响功能 |
| `config` | 配置 | 配置文件修改 |
| `docs` | 文档 | 文档更新 |
| `deps` | 依赖 | 添加/更新/删除依赖 |
| `db` | 数据库 | 数据库结构变更 |
| `security` | 安全 | 安全相关修改 |

## 示例

### 示例 1：Bug 修复

```markdown
# 2026-03-11 开发日志

## [bugfix] 修复订单查询时积分计算错误的 bug

| 字段 | 值 |
|------|-----|
| 时间 | 14:30 |
| 类型 | bugfix |
| 影响文件 | src/api/orders.js, src/utils/points.js |
| 标签 | # 积分 # 订单 # Bug 修复 |

### 变更描述
修复了订单列表查询时，积分计算逻辑错误的问题。
原代码在计算消费返积分时未考虑配置表中的比例设置，
现在改为从 settings 表读取 points_ratio 配置。

### 变更原因
测试发现订单显示的积分与预期不符，
排查发现是硬编码了 1 元=1 分的逻辑，
改为动态读取配置后问题解决。

### 测试验证
- 手动创建 3 笔不同金额的订单，积分计算正确
- 修改配置表中 points_ratio 为 2，验证积分翻倍
```

### 示例 2：配置修改

```markdown
# 2026-03-11 开发日志

## [config] 修改密钥生成逻辑，从 27 位改为 32 位

| 字段 | 值 |
|------|-----|
| 时间 | 10:15 |
| 类型 | config |
| 影响文件 | src/auth.js, .env, .env.example |
| 标签 | # 认证 # 密钥 # 安全 |

### 变更描述
将 JWT 密钥从 27 位自定义字符串改为 32 位标准格式，
以符合安全最佳实践。

### 变更原因
之前使用的 27 位密钥长度不规范，
改为 32 位可以兼容更多加密库，
且符合 OWASP 推荐的密钥长度。

### 测试验证
- 使用新密钥重新生成测试用户 token
- 验证 token 解析正常
- 验证旧 token 已失效（预期行为）
```

### 示例 3：新功能

```markdown
# 2026-03-10 开发日志

## [feature] 添加商品上下架功能

| 字段 | 值 |
|------|-----|
| 时间 | 16:45 |
| 类型 | feature |
| 影响文件 | src/api/products.js, src/db/products.js |
| 标签 | # 商品 # 管理后台 # 新功能 |

### 变更描述
在商品管理 API 中添加上下架功能：
- PUT /admin/products/:id/toggle-sale
- 切换 is_on_sale 字段
- 返回更新后的商品状态

### 变更原因
管理后台需要控制商品是否对用户可见，
上下架功能可以在不删除商品的情况下临时隐藏商品。

### 测试验证
- 调用接口上架商品，查询确认 is_on_sale=true
- 调用接口下架商品，商品列表不再展示
- 商品详情接口返回 404（下架商品）
```

## 最佳实践

1. **及时性**：变更发生后立即记录，避免遗忘
2. **准确性**：文件列表使用 git diff 或手动确认
3. **完整性**：Bug 修复需说明原因和验证方式
4. **标签化**：合理打标签便于后续检索
5. **可追溯**：重要变更关联 Issue ID 或需求编号
